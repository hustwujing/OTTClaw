==============================
skill_id: skill_creator
name: Skill Creator
display_name: Skill Workshop
enable: true
description: Helps users design and create new skills (SKILL.md + optional scripts + optional reference files + optional assets) through interactive dialogue, with hot-reload — no service restart required
trigger: When the user wants to create a new skill, add custom functionality, design a new workflow, or extend system capabilities
==============================

## Skill Goal

Guide the user through a conversation to design and write a complete skill package starting from a requirements description:
- **SKILL.md** (required): skill metadata + detailed workflow
- **script/*.py etc.** (optional): Python / Shell script skeletons, used with `skill(action=run_script)`
- **references/*.md etc.** (optional): reference documents, style guides, templates, spec docs, used with `skill(action=read_reference)`
- **assets/** (optional): asset files such as images, data files, binary resources, used with `skill(action=read_asset)`

Once complete, hot-reload into the system for immediate use.

---

## Execution Steps

### Step Zero: Read the Format Template

Call `skill(action=read_asset, skill_id=skill_creator, asset_name=skill_template.md)` to retrieve the SKILL.md format specification and Python script skeleton template. Follow them strictly throughout the subsequent steps.

---

### Step One: Collect Basic Information

Ask the user the following questions one at a time (wait for each answer before continuing):

1. **Skill purpose** (open-ended): "What do you want this skill to do? Please describe the core functionality or use case."
2. **skill_id**: "Please provide an ID for the skill — only lowercase letters, numbers, and underscores are allowed (e.g. `data_cleaner`, `email_sender`)."
   - If the user provides a non-conforming ID, explain the constraints and ask them to try again.
3. **name** (skill name): "What is the name of the skill? (e.g. 'Data Cleaner')"
4. **display_name** (conversation display name): "What name should the AI display in the conversation interface when this skill is active? This is the speaker identity the user sees — keep it concise and recognizable (e.g. 'Data Analyst', 'Reviewer'). If left blank, `name` is used automatically."
5. **trigger** (trigger condition): "What should the user say to trigger this skill?" (Help the user distill this into a single sentence, e.g. "When the user wants to clean or organize data")

> The `description` field is auto-generated by you based on the skill's purpose — no need to ask the user for it.

---

### Step Two: Confirm Whether Scripts Are Needed

Ask the user: "Does this skill need to execute Python scripts to perform data processing, external API calls, format conversion, or similar tasks?"

- If **scripts are needed**:
  - Ask for the script file name (e.g. `process.py`)
  - Ask for the script's specific functionality: what input it receives (command-line arguments) and what it outputs (stdout)
  - Generate a complete Python script skeleton based on the described functionality (follow the skeleton format in the template)
  - If multiple scripts are needed, repeat the above process

- If **not needed**: skip this step

---

### Step Three: Confirm Whether Reference Files or Assets Are Needed

**Part A — Reference files** (`references/`, read with `skill(action=read_reference)`):

Ask the user: "Does this skill need to read any reference documents (such as style guides, templates, spec docs, glossaries, or example files)?"

- If **reference files are needed**:
  - Ask for the file name (e.g. `style_guide.md`, `prompt_template.txt`)
  - Ask what content the file should contain
  - Generate the complete file content
  - Repeat for multiple files

- If **not needed**: skip Part A

**Part B — Asset files** (`assets/`, read with `skill(action=read_asset)`):

Ask the user: "Does this skill need any asset files (such as images, data files, or binary resources)?"

- If **asset files are needed**:
  - Ask for the file name and content/description
  - Generate or record accordingly
  - Repeat for multiple files

- If **not needed**: skip Part B

---

### Step Four: Design the Skill Content (CONTENT)

Based on the user's described purpose, **draft a complete workflow for the CONTENT section**, which should include:

- **Skill goal**: a paragraph explaining what problem the skill solves
- **Execution steps** (Step One, Step Two, ...): a clear, step-by-step process
  - If there are scripts: specify which step calls `skill(action=run_script)` and what arguments to pass
  - If there are reference files: specify which step calls `skill(action=read_reference)` and which file to read
  - If there are asset files: specify which step calls `skill(action=read_asset)` and which file to read
  - When user interaction is needed: specify when to use `notify(action=options)` or `notify(action=confirm)`
  - When data needs to pass between steps: specify when to use `kv(action=set, ...)` / `kv(action=get, ...)`
  - **No-script warning (mandatory when no scripts)**: if the skill has no script files, insert the following block immediately before the first execution step:
    ```
    > ⚠️ **Execution Mode:** This skill has no script files. All steps are executed directly by the LLM using built-in tools (read_file, kv, etc.). **Never call `skill(action=run_script)`.**
    ```
  - **Sequential execution notice (mandatory in ALL generated skills)**: insert this line at the very start of the Execution Steps section, before Step One:
    ```
    > Execute all steps strictly in order, one step at a time. Do not skip or merge steps. Wait for each step's result before proceeding to the next.
    ```
- **Output format**: describe the form of the final output (text, table, JSON, etc.)
- **Notes** (optional): edge cases, error handling suggestions

After drafting, present the complete draft to the user and ask if any changes are needed. Iterate until the user is satisfied.

**Once the user is satisfied with everything**, perform English translation (Step Four Point Five), then trim for token efficiency (Step Four Point Six, which also saves to KV), then proceed to Step Five.

---

### Step Four Point Five (must do): Translate to English

Before saving to KV, translate all file content to English to reduce token costs on every subsequent invocation.

**Content that must be translated**:

- **SKILL.md HEAD section**: the values of `name`, `display_name`, `description`, and `trigger` fields (`skill_id` stays unchanged, as it is a technical identifier)
- **SKILL.md CONTENT section**: all body text (headings, step descriptions, notes, etc.)
- **All script files**: code comments, print output, error messages, etc.

**Content to leave unchanged**:

- `skill_id` field value
- The code logic itself (only translate comments and string messages, do not alter logic)
- Markdown formatting symbols
- Tool names (`skill`, `kv`, etc.) and parameter names
- **Reference files and asset files**: these files contain example/reference materials that must strictly follow the user's own language and format preferences — do NOT translate them

After translation, replace the original draft with the translated version. No need to show the diff to the user — proceed directly to Step Four Point Six.

---

### Step Four Point Six (must do): Trim SKILL.md for Token Efficiency

SKILL.md CONTENT is loaded on every invocation — every unnecessary word permanently increases per-use cost. After translation, rewrite only the SKILL.md CONTENT to minimize length while keeping all functional instructions intact.

**Apply these rules:**

- **Remove meta-commentary**: delete sentences that explain what the next sentence says (e.g. "In this step, we will…").
- **Use direct imperatives**: replace "You should do X in order to achieve Y" with "Do X."
- **Remove hedging**: delete "if possible", "try to", "note that", "keep in mind that", "remember that".
- **Trim examples**: keep at most one minimal example per concept; remove examples that merely repeat what the instruction already says.
- **Collapse bullets**: if a bullet point has only one sub-item, merge them into a single line.
- **No duplication**: remove any content already covered in Notes, reference files, or asset files.

Do not trim script files, reference files, or asset files — only the SKILL.md CONTENT section.

After trimming, save all content to KV and immediately proceed to Step Five:

```
kv(action=set, key="_draft_skill_md",    value=<complete SKILL.md text — translated and trimmed>)
kv(action=set, key="_draft_scripts",     value=<JSON array, each item {"name":"...", "content":"..."} (content translated), pass [] if no scripts>)
kv(action=set, key="_draft_references",  value=<JSON array, each item {"name":"...", "content":"..."} (content kept in original language, NOT translated), pass [] if no reference files>)
kv(action=set, key="_draft_assets",      value=<JSON array, each item {"name":"...", "content":"..."} (content kept in original language, NOT translated), pass [] if no asset files>)
```

---

### Step Five: Send Creation Confirmation

⚠️ **This step must end with a `notify(action=confirm)` tool call — it is absolutely forbidden to ask the user "please confirm" or "any changes?" in plain text, as doing so will prevent Step Six from ever executing.**

List the files about to be created (file names only, no need to show full content), then **immediately call `notify(action=confirm)`** with the message:

```
About to create skill "{name}" (skill_id: {skill_id}), including:
- SKILL.md
- {list of script files, if any, one per line}
- {list of reference files, if any, one per line}
- {list of asset files, if any, one per line}

Upon confirmation, files will be written and hot-loaded. This cannot be automatically overwritten after writing.
```

- Confirm label: "Confirm Creation"
- Cancel label: "Go Back and Edit"

If the user selects "Go Back and Edit", return to Step Four to continue editing (after editing, re-run `kv(action=set, ...)` to save the latest content, then call `notify(action=confirm)` again).

---

### Step Six: Write Files and Hot-Reload

**Trigger: the user selected "Confirm Creation" in the `notify(action=confirm)` from Step Five.**

First retrieve the draft content saved in Step Four via `kv(action=get, ...)`, then write in the following order, using `notify(action=progress)` to report progress before each step:

1. `kv(action=get, key="_draft_skill_md")` — retrieve SKILL.md content
2. `notify(action=progress)`: "Validating and writing SKILL.md..."
3. `skill(action=write, skill_id=..., content=<content retrieved from kv>)`
   - The tool will automatically validate the format (two separators, required fields, non-empty CONTENT)
   - If a format error is returned, show the specific error to the user and return to Step Four to fix it
   - If an "already exists" error is returned, ask the user whether to use a different skill_id
4. `kv(action=get, key="_draft_scripts")` — retrieve script list (skip if empty array)
5. (If scripts exist) `notify(action=progress)`: "Writing script files..."
6. (If scripts exist) For each script in the list, call `skill(action=write, skill_id=..., content=..., sub_path="script/<script_name>")`
7. `kv(action=get, key="_draft_references")` — retrieve reference file list (skip if empty array)
8. (If reference files exist) `notify(action=progress)`: "Writing reference files..."
9. (If reference files exist) For each file in the list, call `skill(action=write, skill_id=..., content=..., sub_path="references/<reference_name>")`
10. `kv(action=get, key="_draft_assets")` — retrieve asset file list (skip if empty array)
11. (If asset files exist) `notify(action=progress)`: "Writing asset files..."
12. (If asset files exist) For each file in the list, call `skill(action=write, skill_id=..., content=..., sub_path="assets/<asset_name>")`
13. `notify(action=progress)`: "Hot-reloading skill list..."
14. `skill(action=reload)`
15. `notify(action=progress)`: "Skill '{name}' has been created and loaded successfully!"
16. **Output a plain text message (no tool call)**: "Skill '{name}' (skill_id: {skill_id}) has been created and loaded successfully! Files written: [list all written files]. You can start using it right away."

---

## Notes

- **Confirmation must go through a tool — plain text is forbidden**: the user confirmation in Step Five can only be completed via the `notify(action=confirm)` tool. Never use plain text in Step Five such as "any changes?" or "please confirm if everything looks good" — doing so will cause the conversation round to end immediately after the user replies, making Step Six impossible to execute.
- **KV is the bridge for content handoff**: Steps Four through Six span a conversation turn (because `notify(action=confirm)` interrupts the Agent loop). KV is the only mechanism that can preserve draft content between turns. If `kv(action=set, ...)` is skipped, Step Six will have no file content to retrieve.
- **Format is auto-validated by the tool**: `skill(action=write)` parses the content before writing. Format errors return a descriptive error message — no need to manually verify separator counts.
- **Do not write files without user confirmation**: writing tools must only be called after "Confirm Creation" is received from the `notify(action=confirm)` in Step Five.
- **Tool references in SKILL.md must be consistent**: if CONTENT mentions `skill(action=run_script)`, there must be a corresponding script; if it mentions `skill(action=read_reference)`, there must be a corresponding file in `references/`; if it mentions `skill(action=read_asset)`, there must be a corresponding file in `assets/`.
- **Python script skeleton quality**: scripts should include an entry function, argument parsing, and error handling, following the skeleton format in the template, so that users can quickly fill in the business logic.
- **CONTENT steps must be specific**: each tool call should clearly specify skill_id and parameter names — avoid vague descriptions like "call some tool".
- **SKILL.md must be concise**: SKILL.md CONTENT is loaded on every skill invocation. Keep all descriptions brief and actionable — omit verbose explanations and redundant examples. Every extra sentence increases token cost each time the skill runs.
